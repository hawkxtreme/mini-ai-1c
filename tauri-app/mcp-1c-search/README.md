# 1c-search MCP Server

Высокопроизводительный MCP-сервер для поиска и навигации по исходным кодам конфигураций 1С:Предприятие (выгрузка в файлы). Оптимизирован для работы с огромными конфигурациями (20ГБ+, 100к+ файлов) даже на медленных HDD.

## 🚀 Возможности

- **Мгновенный поиск символов**: Поиск определений процедур и функций по всей конфигурации за миллисекунды.
- **Интеллектуальный Grep**: Полнотекстовый поиск по `.bsl` и `.xml` файлам с поддержкой регулярных выражений и ограничением области поиска (`scope`).
- **Анализ метаданных**: Исследование структуры конфигурации (справочники, документы, реквизиты, формы) через парсинг `Configuration.xml` и `ConfigDumpInfo.xml`.
- **Поиск ссылок и анализ влияния**: Быстрое нахождение всех вхождений символа в коде.
- **Инкрементальный индекс**: Автоматическое обновление индекса только для измененных файлов на основе `mtime`.

## 🏗 Архитектура и Алгоритмы

### 1. Индексация символов (Symbol Indexing)
*   **Точный парсинг**: Используется инкрементальный парсер **Tree-sitter** для извлечения AST из BSL-кода. Это позволяет точно определять границы процедур, их имена и статус `Экспорт`, игнорируя комментарии и строковые литералы.
*   **Хранение**: Данные сохраняются в SQLite с использованием B-Tree индексов, что обеспечивает мгновенный поиск даже при миллионах записей.

### 2. Умный поиск (Search Logic)
*   **Двухфазный поиск по символам**:
    *   *Фаза 1 (Prefix Match)*: Поиск по началу имени. Использует индекс SQLite (O(log n)), работает мгновенно.
    *   *Фаза 2 (Substring Match)*: Выполняется только если нужно добрать результаты. Полный скан имен внутри БД.
*   **Index-Guided Code Search**: При полнотекстовом поиске система сначала опрашивает индекс символов, чтобы найти "горячие" файлы (где эти символы объявлены). Эти файлы проверяются первыми, что позволяет выдать результат до завершения полного сканирования.

### 3. Граф метаданных (Metadata Graph)
*   Сервер строит иерархическую модель конфигурации, позволяя ИИ понимать связи между объектами (например, какие реквизиты принадлежат конкретной табличной части документа).

## ⚡ Оптимизации

1.  **Инкрементальная синхронизация**: SQLite хранит время модификации (`mtime`) каждого файла. При обновлении перепарсиваются только измененные модули.
2.  **Максимальный параллелизм**: Парсинг файлов (чтение + построение AST) распараллелен через библиотеку **Rayon** на все доступные ядра процессора.
3.  **Streaming & Early Exit**: Поиск в файлах идет потоком и останавливается сразу, как только набрано нужное количество результатов (`limit`).
4.  **Time Budgeting (Таймауты)**: Операции поиска ограничены жестким дедлайном (8 секунд) для предотвращения зависаний на слабых системах.
5.  **SQLite WAL Mode**: Использование режима Write-Ahead Logging позволяет выполнять чтение и запись в индекс одновременно без блокировок.

## 🛠 Технологический стек

- **Rust**: Ядро сервера для максимальной производительности.
- **SQLite (rusqlite)**: Надежное и быстрое хранилище индексов.
- **Tree-sitter (tree-sitter-bsl)**: Промышленный стандарт парсинга исходного кода.
- **Ignore (WalkBuilder)**: Быстрый рекурсивный обход папок с учетом правил игнорирования.

## ⚙️ Настройка

Сервер настраивается через переменные окружения:

- `ONEC_CONFIG_PATH`: Абсолютный путь к директории с исходниками конфигурации.
